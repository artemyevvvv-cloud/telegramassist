Клиент заплатил. Деньги списались с его счета. Но твой сайт об этом пока не знает. Сейчас настроим автоматическое уведомление сервера о завершении платежа.

## [ КАК РАБОТАЕТ ВЕБХУК ]

После успешной оплаты, NowPayments отправляет специальное сообщение на твой сервер:

```text
Сообщение приходит на адрес: https://твой-сайт.com/api/webhook

Внутри сообщения:
- x-nowpayments-sig: "abc123def456..."  ← Цифровая подпись (печать подлинности)
- Данные платежа:
  {
    "id": 123,
    "status": "finished",           ← Статус: оплата завершена
    "order_id": "order_789",        ← Номер заказа
    "pay_amount": 50,               ← Сумма платежа
    "price_currency": "usd"         ← Валюта
  }
```

Наша задача:

1. Получить этот запрос на сервере

1. **Проверить подпись** (убедиться что запрос действительно от NowPayments)

1. Если статус = `'finished'` → выдать товар пользователю.

---

## [ ПРОВЕРКА ПОДПИСИ ]

**Проблемка, бро: **щас любой продвинутый пользолователь может отправить фейковое сообщение на твой сайт, притворившись NowPayments. Если ты не проверишь, что это действительно NowPayments — выдашь товар бесплатно. **Решение: цифровая подпись.**

Для упрощения, представь, что NowPayments ставит печать на каждое сообщение. Эту печать может поставить только тот, кто знает секретное слово (`IPN Secret`). Только ты и NowPayments знаете это секретное слово.

### **Как работает проверка:**

1. **NowPayments отправляет сообщение:**

- Берет текст сообщения (например: `{"status":"finished","order_id":"789"}`)

- "Печатает" его секретным словом (`IPN Secret`)

- Получается уникальная печать (длинная строка букв и цифр)

- Отправляет сообщение + печать

1. **Твой сервер проверяет:**

- Получает сообщение + печать

- Берет то же секретное слово (`IPN Secret`)

- "Печатает" сообщение сам

- Сравнивает свою печать с печатью от NowPayments

1. **Результат:**

- Печати совпали → это точно NowPayments, можно доверять

- Печати НЕ совпали → это мошенник, игнорируй сообщение

Такая "печать" называется **HMAC подпись** (математический способ проверки подлинности). Тебе не обязательно понимать как она работает внутри, библиотеки сделают всё за тебя.

**Технический нюанс в Vercel:**

Когда сообщение приходит на сервер, Vercel автоматически "обрабатывает" текст (превращает его в удобный формат). Но для проверки подписи нужен **исходный текст "как есть"** (без обработки).

**Решение:** попроси Claude написать код, который получает "сырой текст" сообщения для проверки подписи.**Типа такого:**

```text
Создай Vercel Function api/webhook.ts для обработки NowPayments webhook.

Требования:
1. Получи raw body (сырой текст запроса, не распарсенный JSON)
2. Вычисли HMAC SHA512 подпись от raw body используя IPN_SECRET
3. Сравни с заголовком x-nowpayments-sig
4. Если подпись НЕ совпадает → верни 403 Forbidden
5. Если статус = 'finished' → обнови пользователя в Supabase
6. Используй SUPABASE_SERVICE_ROLE_KEY для обновления базы
```

Ну или можно просто по человечески ему объяснить, что тебе надо защитить свой проект от подделки платежей. Он поймёт.

## [ SERVICE ROLE KEY: КЛЮЧ АДМИНА ]

В Supabase есть два типа ключей:

1. **Обычный ключ (`anon`)** — для пользователей сайта

- Каждый пользователь видит только свои данные

- Не может изменить данные других людей

1. **Ключ администратора (`service_role`)** — для сервера

- Видит все данные

- Может изменить любую запись в базе

**Почему тут нужен ключ администратора:**

Webhook приходит от NowPayments (не от конкретного пользователя). Он должен обновить статус в базе: `is_pro = true`. Если использовать обычный ключ — база откажет в доступе.

Короче, в webhook используй специальный ключ `SUPABASE_SERVICE_ROLE_KEY`, которому база доверяет полностью.

### **Где взять:**

1. Supabase Dashboard → Settings → API

1. Скопируй `service_role` ключ

1. Добавь в `.env`:

```text

SUPABASE_SERVICE_ROLE_KEY=eyJ...

```

1. Добавь в Vercel Environment Variables (Settings → Environment Variables)

### **Как использовать:**

Claude уже знает как работать с Service Role Key. В промпте для создания webhook мы указали `SUPABASE_SERVICE_ROLE_KEY` — Claude сам напишет код правильно.

Если хочешь проверить что Claude сделал всё верно, попроси его:

_"Покажи где в коде webhook используется SUPABASE_SERVICE_ROLE_KEY и объясни зачем он нужен"_

**⚠️ КРИТИЧЕСКОЕ ПРЕДУПРЕЖДЕНИЕ:**

❌ **НИКОГДА не используй Service Role Key в браузере**

❌ **НИКОГДА не коммить его в git**

✅ Используй только в серверных функциях (API routes, Edge functions)

✅ Храни только в переменных окружения

_Если Service Role Key попадет в публичный код — злоумышленник получит полный доступ к твоей базе данных. А это будет пиздос._

## [ СТАТУСЫ ПЛАТЕЖА ]

NowPayments отправляет webhook НЕСКОЛЬКО раз, при каждом изменении статуса платежа. Твой код получит несколько сообщений по одному платежу:

**| Статус | Что происходит | Что делать |**

| `waiting` | Клиент открыл страницу оплаты, но не отправил деньги | Жди |

| `confirming` | Деньги отправлены, идет проверка в сети (занимает время) | Жди |

| `sending` | Деньги подтверждены, сейчас переведут на твой кошелек | Жди |

| `finished` | **Всё! Деньги у тебя.** | **ВЫДАЙ ТОВАР** |

| `failed` | Что-то не так, платеж не прошел | Покажи ошибку |

**⚠️ САМОЕ ВАЖНОЕ:
**Выдавай товар **ТОЛЬКО** на статусе `finished`. Если выдашь на `waiting` или `confirming` — клиент может не заплатить до конца, а товар уже получит бесплатно.

## [ ЧАСТЫЕ ОШИБКИ ]

### 1. "Подпись не совпадает" (код возвращает 403 Forbidden)

**Что произошло****:** твоя "печать" не совпала с печатью от NowPayments.

**Возможные причины****:**

- Скопировал неправильный `IPN_SECRET` из NowPayments (проверь еще раз)

- Код проверяет "обработанный текст" вместо "сырого текста" (см. раздел про Vercel выше)

- В коде есть лишние пробелы или переносы строк

**Как исправить****:** ропроси Claude проверить код проверки подписи, покажи ему ошибку.

### 2. База данных не обновляется

**Что произошло****:** Webhook получил сообщение, но статус пользователя не изменился (`is_pro` остался `false`).

**Причина****:** используешь обычный ключ (`anon`) вместо ключа администратора.

**Как исправить****:** проверь что в коде используется `SUPABASE_SERVICE_ROLE_KEY`, а не обычный ключ.

### 3. Товар выдается до оплаты

**Что произошло****:** клиент получил доступ к товару, но деньги еще не пришли.

**Причина****:** код выдает товар на статусе `waiting` или `confirming` (когда оплата еще не завершена).

**Как исправить****:** проверь что товар выдается ТОЛЬКО когда `status === 'finished'`.

💭 **От Тома:
**Webhook это самая критичная часть платежной интеграции.

**Если допустишь тут ошибку, потеряешь деньги:
**- Забыл проверить подпись → выдашь товар бесплатно
- Выдал товар на 'waiting' → клиент получит товар без оплаты
- Используешь Service Role Key в браузере → взломают базу данных

Потрать на эту часть столько времени, сколько нужно. Проверь 10 раз. Протестируй на реальном платеже в тестовом режиме. Когда все работает, можно спать спокойно.
