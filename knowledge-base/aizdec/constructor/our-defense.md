# Три цвета риска 🌈

Каждое действие агента имеет уровень риска. Ты это видишь в работе, даже если не задумываешься.

**🟢 GREEN — агент делает сам, молча.**

Прочитать файл. Поискать в интернете. Найти что-то в памяти. Безопасные операции, которые ничего не меняют. Ты просто получаешь результат.

**🟡 YELLOW — агент делает, но показывает что сделал.**

Записать файл. Сохранить заметку. Выполнить команду в терминале. Git-коммит. Операции, которые меняют состояние. Агент выполняет, но ты видишь в ответе что именно он изменил.

**🔴 RED — агент останавливается и спрашивает.**

`rm -rf`, `git push --force`, `DROP TABLE`, `shutdown`, деплой на продакшен. Ты видишь кнопки ✔ / ✖ в Telegram. Пока не нажмёшь — агент ждёт.

Ключевой момент: уровень определяется не названием инструмента, а конкретной командой. `git status` — жёлтый. `git push --force` — красный. `ls` — жёлтый. `rm -rf /` — красный. Система анализирует содержимое команды.

## Попробуй сам

Лучший способ понять защиту = протестировать. Открой Telegram и попробуй:

**"Покажи содержимое файла .env"**

→ Агент откажет. Файл в списке запрещённых.

**"Выполни команду sudo apt update"**

→ Заблокировано. `sudo` запрещено в любом месте строки.

**"Выполни printenv"**

→ Заблокировано. Команды для просмотра окружения запрещены.

**"Покажи мой GITHUB_TOKEN"**

→ Агент не покажет значение. Даже если выполнит команду, которая его содержит — значение будет заменено на `[REDACTED]`.

**"Удали всё в домашней директории"**

→ Агент покажет кнопки ✔ / ✖ и будет ждать. Не нажимай ✔.

**"Запиши файл в /etc/test.txt"**

→ Не сможет. Systemd «sandbox» разрешает запись только в workspace/ и projects/.

**Добавь тестовую переменную** через /settings (например `TEST_KEY` со значением `hello12345`). Попроси агента `echo $TEST_KEY`. Значение не появится в чате — система отфильтрует автоматически. Это dynamic sanitization: любое значение из /settings попадает в фильтр в реальном времени.

Все эти ограничения работают независимо от того что ты написал агенту. Даже если через «prompt injection» кто-то убедит агента попробовать = технические блокировки сработают.

## Когда нажимать ✔, а когда ✖

**Агент показал кнопки = значит, операция красная. Прежде чем нажать:**

**[1]** Прочитай что агент собирается сделать. Он пишет конкретно: "Хочу выполнить `git push origin main`" или "Хочу удалить файл X".

**[2]** Если понимаешь что это и зачем — ✔.

**[3]** Если не уверен — ✖. Ничего не сломается, агент просто остановится. Можешь спросить "объясни зачем тебе это" и решить.

**[4]** Если агент просит что-то странное (удалить системные файлы, отправить данные куда-то, выполнить незнакомую команду) — ✖ и разберись. Это может быть результат «prompt injection» через текст, который ты переслал.

# Что на автомате, а что нет

**Автоматически (ты не настраиваешь):**

- Системные ключи скрыты от агента

- Файлы `.env`, `.ssh/`, код бота — заблокированы для чтения

- Опасные команды (`sudo`, `kill`, `printenv`) — заблокированы

- Значения ключей фильтруются из ответов в чате

- Файлы личности очищаются от фейковых тегов при загрузке

- Процесс бота ограничен «sandbox» на уровне ОС

**Зависит от тебя:**

- Какие права у токенов (полный доступ vs ограниченный)

- Какие сервисы подключены (меньше = безопаснее)

- Что пересылаешь агенту (ненадёжный текст = риск)

- Когда нажимаешь ✔ на красных операциях

---

# Безопасный режим, следующий шаг

Всё описанное выше = стандартный режим. Ключи сервисов хранятся на VPS, агент использует их для работы, система фильтрует значения из ответов.

В следующем обновлении появится **безопасный режим**. Ключевое отличие: ключи вообще не хранятся на сервере. Отдельный защищённый процесс держит их в оперативной памяти и подставляет только на время выполнения команды. Агент знает какие сервисы подключены, но физически не имеет доступа к значениям ключей.

**Переключение через /settings одной кнопкой. **Текущие ключи автоматически мигрируют в защищённое хранилище. Стандартный режим остаётся для тестовых ключей и быстрой настройки. Безопасный = для продакшена.
