# Система памяти

**Курс:** constructor  
**URL:** https://www.aizdec.me/courses/constructor/memory

---

Агент записывает факты из разговоров, ведёт дневник, сохраняет статьи в базу знаний, отслеживает задачи. Часть этого происходит автоматически, часть — когда ты просишь.

**Чтобы использовать память эффективно, **нужно понимать как она устроена. Не на уровне _"есть файлы",_ а на уровне механики: что загружается в контекст, почему именно так, и как это влияет на работу агента.

## Почему несколько уровней, а не один файл

Контекстное окно модели ограничено. Каждый раз когда ты пишешь агенту, он загружает в запрос: SOUL.md, USER.md, MEMORY.md, CLAUDE.md, сегодняшний дневник, каталог базы знаний, активные навыки, LEARNED.md, состояние системы. Всё это = контекст. Что не в контексте = агент не помнит, пока не найдёт через поиск.

Если свалить всё в один файл = он не влезет или вытеснит место для самого разговора. Поэтому память разделена на уровни с разной скоростью доступа:

| Уровень | Что хранит | Когда загружается | Размер |
| --- | --- | --- | --- |
| **MEMORY.md** | Постоянные факты о тебе | Каждое сообщение | До 200 строк |
| **LEARNED.md** | Выводы и паттерны агента | Каждое сообщение | Заполняется агентом |
| **memory/** | Дневник по дням | Сегодня — всегда, 3 дня — по запросу | Без лимита |
| **.current-task.md** | Активная задача | Каждое сообщение | 1 файл |
| **ops/tasks.json** | Трекер задач | По запросу | Без лимита |
| **knowledge/** | Статьи, ресёрчи | Каталог — всегда, статьи — при поиске | Без лимита |

**MEMORY.md — быстрый доступ, но маленький. Дневник — средний. База знаний — огромная, но нужен умный поиск.**

# MEMORY.md — что агент помнит всегда

Загружается в контекст при каждом сообщении. Всё что здесь = агент знает без поиска. Именно поэтому файл должен быть компактным: 200 строк это баланс между полнотой и размером контекста.

**Организован по секциям:**

```markdown
# Память

## Preferences
- Предпочитает таблицы для отчётов
- Не любит длинные вступления

## Projects
- SaaS для аналитики, стек: Next.js + Supabase
- Telegram-бот для клиентов, Python

## Contacts
- Дизайнер: Алексей, @alex_design
```

**Что сюда попадает: **предпочтения, активные проекты, стек, контакты, ключевые решения. Одна-две строки на факт.

Жёсткий лимит в коде — 10 КБ. При 8 КБ агент получает предупреждение в контексте и начинает консолидацию. Встроенная дедупликация: агент проверяет первые 60 символов нового факта — если похожее уже есть, не добавит повтор.

**Когда агент записывает в MEMORY.md:**

- Ты рассказал о себе — имя, профессия, стек, предпочтения

- Принял ключевое решение по проекту

- Повторяющийся паттерн (спросил одно и то же 3+ раз — агент запишет)

- Новый контакт, URL, конфигурация которые нужны постоянно

# LEARNED.md — что агент понял сам

Загружается в контекст как** "Что я узнал"**. Агент заполняет после сессий: какие паттерны заметил, какие решения сработали, какие подходы ты предпочитаешь. Ты этот файл обычно не трогаешь — но можешь открыть и посмотреть что агент о тебе "думает".

# memory/ — дневник

Файл на каждый день: `memory/2026-03-16.md`. Агент загружает сегодняшний дневник автоматически. Предыдущие 3 дня проверяет если нужен контекст. Можно глубже, если попросишь.

```markdown
## 14:22 — Деплой бота
- Настроил GitHub Actions CI/CD
- Интеграция с Vercel

## 16:45 — Баг: auth loop
- Проблема: тайминг обновления токена
- Решение: debounce на login
```

**Зачем отдельно от MEMORY.md:** разный жизненный цикл. MEMORY.md — факты на месяцы. Дневник — что произошло сегодня. Если бы всё было в одном файле, он разросся бы за неделю.

**Когда агент записывает в дневник:**

- После выполнения задачи — что сделали, ключевые решения

- При смене темы — итог предыдущей

- Перед завершением диалога — ключевые выводы

- Когда контекст становится большим — сбрасывает важное в дневник, чтобы не потерять между сессиями (memory flush)

# .current-task.md — активная задача

**Для многошаговых задач (5+ шагов) агент создаёт файл** `.current-task.md` в workspace: план, текущий прогресс, следующий шаг. **Загружается автоматически при каждом сообщении.**

**Зачем: **если сессия прервалась (таймаут, лимит, рестарт) — агент при следующем сообщении сразу видит где остановился и продолжает. Без этого файла после рестарта он забудет что делал.

Когда задача завершена = агент удаляет файл.

# ops/tasks.json — трекер задач

**Персистентный список задач. Переживает рестарты, не теряется между сессиями.**

→ "Создай задачу: подготовить презентацию к пятнице"

→ "Что у меня висит?"

→ "Отметь задачу X как выполненную"

Каждая задача получает ID. Для проектов с несколькими параллельными задачами — удобнее чем держать всё в голове.

# knowledge/ — база знаний

**Статьи, ресёрчи, архитектурные решения, шаблоны. Организована по темам (папкам):**

```text
knowledge/
  architecture/
    microservices.md
    database-design.md
  deployment/
    docker.md
    vercel-nextjs.md
  _index.md              ← автоматический каталог
```

Сами статьи в контекст не загружаются — слишком большие. Но каталог `_index.md` загружается: агент видит список тем и заголовков без чтения самих статей. Когда нужна конкретная статья — находит через поиск.

Именно поэтому размер не ограничен: можешь сохранять сколько угодно, это не влияет на скорость обычных ответов.

**Хорошие кандидаты для knowledge/:**

- Архитектурные решения ("почему выбрали Supabase, а не Firebase")

- Решения проблем, которые встречались не раз

- Конспекты важных статей и видео

- Шаблоны конфигураций

- Ресёрч по теме

# Как агент решает куда сохранять

**Правила заложены в конфигурации:**

| Ситуация | Куда | Инструмент |
| --- | --- | --- |
| Узнал постоянный факт о тебе | MEMORY.md | `update_memory` |
| Выполнил задачу, принял решение | Дневник | `save_note` |
| Начал многошаговую задачу (5+ шагов) | .current-task.md | `write_file` |
| Длинный материал, ресёрч, шаблон | knowledge/ | `kb_save` |
| Спросили одно и то же 2+ раз | knowledge/ | `kb_save` |
| Контекст перегружен | Дневник (memory flush) | `save_note` |

# Самодиагностика

Агент следит за состоянием памяти автоматически. При каждом сообщении проверяет:

- **MEMORY.md > 8 КБ** → предупреждение: "MEMORY.md разросся. Убери неактуальное, перенеси детали в knowledge/"

- **Контекст близок к лимиту** (32 сообщения в сессии) → "Вызови update_memory — сохрани ключевые факты до автоматической обрезки"

- **USER.md неполный** (меньше 3 из 5 ключевых полей) → агент получает задание собрать недостающую информацию в первых диалогах, естественно, в контексте задач

- **Drift detection** → если агент спрашивает то, что уже записано в MEMORY.md — это сигнал перечитать контекст

**Ты это не видишь = агент получает эти сигналы в системном промпте и действует.**

# Как работает поиск

**Текстовый.** По ключевым словам. Ищет в MEMORY.md, дневнике и базе знаний. Как Ctrl+F — точные совпадения. Быстро, бесплатно, всегда работает.

→ "Найди в заметках упоминания Railway"

**Семантический.** Текст превращается в числовой вектор (через OpenRouter). Поиск = сравнение векторов по близости. «Деплой» и «развёртывание» = разные слова, но близкие векторы. Поэтому находит по смыслу, а не по буквам.

→ "Что я знаю про маркетинг?" — найдёт статьи где слово «маркетинг» не встречается, но тема та же

Семантический поиск работает по умолчанию — дополнительных ключей не нужно. Данные мониторинга хранятся в отдельной SQLite базе на сервере с тем же типом поиска. Подробнее — в уроке 18.

**Результаты кэшируются:** повторный поиск по тем же файлам не требует нового запроса к API.

# Что просишь ты

→ **"Запомни что клиент Z предпочитает таблицы"**

Добавит в MEMORY.md. Через месяц спросишь "как оформить отчёт для Z" — вспомнит.

→ **"Сохрани этот ресёрч в базу знаний"**

Создаст статью в knowledge/. Появится в каталоге, найдётся через поиск.

→ **"Подведи итог дня"**

Запишет в дневник: что сделано, что в процессе, какие решения приняты.

→ **"Что есть в базе знаний?"**

Покажет все темы и статьи из каталога.

→ **"Пересмотри MEMORY.md"**

Покажет устаревшее, предложит удалить или перенести в knowledge/.

**Помни:** если какая-то информация нужна через неделю или месяц = попроси агента сохранить ее явно. Текущие задачи агент запишет сам.

# Память между проектами

Если используешь ~/projects/ (следующий урок), каждый проект может иметь свою knowledge/. Но MEMORY.md, дневник и LEARNED.md — глобальные, всегда в ~/workspace/. Факты о тебе доступны в любом проекте. Переключил проект — агент всё ещё знает кто ты.

# Полезные привычки

**В конце рабочего дня:
**"Подведи итог" — агент запишет структурированный отчёт в дневник.

**Когда нашёл решение проблемы:
**"Сохрани в базу знаний" — найдётся через поиск через месяц.

**Когда начинаешь новый проект:
**"Запомни: проект X, стек Y, цель Z" — попадёт в MEMORY.md.

**Раз в неделю-две:
**"Пересмотри MEMORY.md" — чистка от устаревшего за минуту.

**Перед важным разговором:
**"Что ты знаешь про проект X?" — проверишь, дополнишь.

💭 **От Тома:
**Тема памяти огромная и точно не уместится в один урок. Сейчас ты изучил текущую конфигурацию и понимаешь механику: что куда записывается, как загружается, зачем разделено на уровни. Дальше мы поговорим о том, как превратить это в полноценный второй мозг — ядро, которое с каждым днём всё лучше знает тебя и твои проекты, модернизируется и развивается. Память это не просто заметки. Это то, без чего невозможно представить эффективного агента, так же как невозможно представить эффективного человека.
